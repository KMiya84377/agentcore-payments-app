# Programmatic Payments: AWSやAIエージェント抜きの一般論

## 1. これは何の話か

Programmatic paymentsは、人間が毎回購入ボタンを押すのではなく、ソフトウェアがルールに従って支払いを行う仕組みです。

たとえば、アプリやAPIクライアントが「このデータを1件だけ使いたい」「この有料APIを1回だけ呼びたい」と判断したときに、その場で少額を支払ってアクセスするような用途です。

## 2. そもそも購入とは何か

購入とは、「何かを受け取る代わりに、相手が価値があると認めるものを渡すこと」です。

```text
買う人:
  欲しいものを受け取る

売る人:
  対価を受け取る
```

この対価が、時代によって変わってきました。

## 3. 支払いの流れ

現代の購入を理解するには、次の流れだけ押さえれば十分です。

```text
物々交換
  ↓
通貨
  ↓
銀行口座・振込
  ↓
カード決済
  ↓
電子マネー・オンライン決済
  ↓
アプリ内課金・サブスク・クラウド課金
  ↓
Programmatic payments
```

もともとは物と物を交換していました。そこから通貨が生まれ、銀行口座の残高を動かすようになり、カード会社を通した決済やオンライン決済が広がりました。

現代の購入はかなりデジタル化されています。ただし、多くの場合はまだ「人間が購入ボタンを押す」ことが前提です。

Programmatic paymentsは、ここからさらに進んで「ソフトウェアが条件を読み取り、支払い、アクセスする」方向の話です。

## 4. 購入対象がデジタル化した

インターネットが広がると、購入対象は物理的な商品だけではなくなりました。

- 音楽
- 電子書籍
- 動画
- ニュース記事
- ゲーム内アイテム
- ソフトウェア
- クラウドストレージ
- オンラインサービスの利用権

ここで重要なのは、買うものが「物」から「データ」や「利用権」に広がったことです。

```text
物理的な商品を買う
  ↓
デジタルコンテンツを買う
  ↓
オンラインサービスの利用権を買う
```

## 5. ソフトウェアが他のソフトウェアを使う

さらに進むと、人間が画面でサービスを使うだけでなく、ソフトウェアが別のソフトウェアの機能を使うようになります。

たとえば、天気アプリは天気データを取得します。ECサイトは配送サービス、決済サービス、住所確認サービスと連携します。社内システムは外部の翻訳、地図、検索、分析サービスを呼び出します。

この「ソフトウェアが別のソフトウェアを呼び出す入口」がAPIです。

```text
人間:
  Webサイトやアプリ画面を操作する

ソフトウェア:
  APIを呼び出して機能やデータを使う
```

## 6. 何が課題なのか

SaaSやクラウドでは、利用量に応じた課金が一般的です。

```text
ストレージを何GB使ったか
APIを何回呼んだか
サーバーを何時間動かしたか
AIモデルを何トークン使ったか
```

これはProgrammatic paymentsに近いですが、多くの場合は事前に契約、アカウント作成、請求先登録が必要です。

一方で、ソフトウェア同士の世界では、もっと細かい購入が起きます。

```text
記事を1本だけ読む
データを1件だけ取得する
APIを1回だけ呼ぶ
分析処理を1回だけ実行する
外部ツールを1回だけ使う
```

このような「今すぐ、1回だけ、少額だけ」の購入では、従来の決済が重くなります。

- クレジットカードは手数料やオーソリ処理が重い
- 月額契約は使わない分まで払うことがある
- 請求書払いはリアルタイムなアクセス許可に向かない
- APIキー課金は事前契約が必要になりやすい

## 7. マイクロペイメントとは

マイクロペイメントは、非常に小さな金額の支払いを指します。

金額の厳密な定義は文脈によって違いますが、数円、数十円、数セント、場合によっては1セント未満の支払いです。

```text
マイクロペイメント:
  支払い金額が小さい、という話

Programmatic payments:
  支払いをプログラムから実行・制御しやすい、という話
```

両者は同じものではありません。ただし、ソフトウェアがAPIやデータを必要な分だけ使う場合、1回あたりの金額は小さくなりやすいため、現実にはかなり重なります。

## 8. Programmatic paymentsとは

Programmatic paymentsでは、プログラムが支払い条件を読み取り、許可された範囲で支払い、アクセスを得ます。

```text
人間が買う
  ↓
アプリが買う
  ↓
APIクライアントやソフトウェア同士が買う
```

Programmatic paymentsという言葉自体は、特定の1社や1団体が定義した単一の標準名ではありません。

似た言葉として、次のような表現があります。

- programmable payments
- machine-to-machine payments
- internet-native payments
- web payments
- micropayments
- API payments

つまり、Programmatic paymentsは「誰かが作った1つの製品名」ではなく、支払いをソフトウェアから扱いやすくする方向性を指す言葉です。

## 9. 支払い手段には何があるか

Programmatic paymentsの支払い手段は、ステーブルコインだけではありません。

代表的な選択肢は次です。

```text
カード決済:
  Stripeなどの決済サービス経由で支払う

銀行振込・口座振替:
  口座間でお金を動かす

請求書・後払い:
  月末などにまとめて請求する

プリペイド残高:
  事前にチャージして、利用時に引き落とす

プラットフォーム内クレジット:
  APIクレジット、AI推論クレジット、ゲーム内通貨など

暗号資産:
  ビットコイン、イーサリアムなど

ステーブルコイン:
  USDCやJPYCのように法定通貨に連動するデジタル資産

Interledger系:
  複数の支払いネットワークをまたぐ価値転送プロトコル
```

人間向けの買い物ではカードや電子マネーが強いです。企業間の大きな取引では請求書や銀行振込が強いです。

一方で、オープンなAPI間の即時・少額・自動決済では、ステーブルコインや暗号資産、Interledger系の仕組みが候補になります。

## 10. ステーブルコインとは

ステーブルコインは、価格が米ドルや日本円などの法定通貨に連動するように設計されたデジタル資産です。

代表例には、米ドルに連動するUSDCや、日本円に連動するJPYCがあります。

```text
USDC:
  1 USDC = 1米ドルを目指す米ドル建てステーブルコイン

JPYC:
  1 JPYC = 1円を目指す日本円建てステーブルコイン
```

ビットコインやイーサリアムのような暗号資産は価格変動が大きいですが、ステーブルコインは価値の安定を目指します。

Programmatic paymentsの文脈では、ステーブルコインは「プログラムから扱いやすいデジタルな支払い残高」として使われます。

なお、JPYCは過去にJPYC Prepaidという前払式支払手段として展開されていた時期があります。現在の日本円建てステーブルコインJPYCとは、法的な位置づけやサービス形態を区別して理解する必要があります。

## 11. なぜステーブルコインが出てくるのか

ステーブルコインが出てくる理由は、少額決済と自動化に向いているからです。

- 小さな金額を扱いやすい
- 国やカード会社をまたいだ支払いをしやすい
- APIやプログラムから支払い処理を組み込みやすい
- 支払い証明を機械的に検証しやすい

ただし、ステーブルコインは唯一の選択肢ではありません。

銀行口座、カード、プリペイド残高、ポイント、独自クレジットでも似た仕組みは作れます。ステーブルコインは、その中でもHTTP/APIレベルの即時・少額・自動決済と相性がよい選択肢の1つです。

## 12. Web3との関係

Web3も関係します。ただし、Programmatic paymentsにWeb3が必須というわけではありません。

ここではWeb3を、ざっくり「ブロックチェーン、暗号資産、ウォレット、スマートコントラクトなどを使うインターネット上の仕組み」と考えます。

Web3を使う場合、次の要素が関係します。

- ウォレット: プログラムやユーザーが支払い元を持つ
- ステーブルコイン: 価格が安定したデジタル資産で支払う
- ブロックチェーン: 支払いの記録や決済を行う台帳になる
- スマートコントラクト: 条件に応じた支払いロジックを実行する
- オンチェーン決済: 銀行やカードネットワークではなくブロックチェーン上で決済する

一方で、Programmatic paymentsはWeb3なしでも作れます。

```text
Web3を使わない例:
  カード決済API
  プリペイド残高
  プラットフォーム内クレジット
  銀行API

Web3を使う例:
  ステーブルコイン決済
  暗号資産ウォレット
  スマートコントラクト決済
  オンチェーンの支払い証明
```

## 13. x402とは

x402は、HTTPの `402 Payment Required` を使って、Web/APIアクセス時に支払いを要求するためのプロトコルです。

通常のWeb/APIでは、アクセス権がなければ `401 Unauthorized` や `403 Forbidden` が返ります。x402では、リソース提供側が「このリソースには支払いが必要」と返し、クライアントは支払い証明を付けて再リクエストします。

```text
1. クライアントが有料APIにアクセスする
2. サーバーが 402 Payment Required を返す
3. クライアントが支払い情報を作る
4. クライアントが支払い証明を付けて再リクエストする
5. サーバーが支払いを検証する
6. 問題なければAPI結果やコンテンツを返す
```

x402は、Coinbaseが開発したオープンな支払いプロトコルです。

```text
広い概念:
  Programmatic payments

具体的なプロトコル:
  x402

x402の開発元:
  Coinbase
```

## 14. 関連する標準・実装基盤

Programmatic payments自体は広い一般概念ですが、関連する標準、プロトコル、実装基盤には具体的な提唱元や提供元があります。

### 14.1 W3C Web Payments

W3Cは、Web上の支払いを標準化するためにWeb Payments Working Groupを持っています。

代表的な仕様にPayment Request APIがあります。これは、Webサイトが複数の支払い方法を扱いやすくし、ブラウザがユーザーと加盟店の間の支払いフローを仲介するためのAPIです。

これは主に「人間がブラウザ上で支払う」ためのWeb標準です。

### 14.2 HTTP 402 Payment Required

HTTPには `402 Payment Required` というステータスコードがあります。

これはデジタルキャッシュやマイクロペイメントのために用意されたものですが、標準的な使い方は定まっていません。x402は、この未活用だった `402 Payment Required` を現代的に使おうとするプロトコルです。

### 14.3 Coinbase

Coinbaseは、暗号資産取引所や開発者向け暗号資産インフラを提供する企業です。

今回の文脈では、2つの形で登場します。

```text
Coinbase:
  x402の開発元

Coinbase CDP:
  Coinbase Developer Platform
  開発者向けにウォレットや暗号資産決済の機能を提供する基盤
```

AWS AgentCore Paymentsでは、Coinbase CDPは外部ウォレット/決済プロバイダーとして再登場します。

### 14.4 Stripe

Stripeは、オンライン決済、請求、サブスクリプション、決済インフラを提供する企業です。

通常の文脈では、クレジットカード決済や銀行決済など、Webサービスの決済基盤として使われます。

今回の文脈では、ステーブルコインや暗号資産決済を扱う支払いレール側の文脈で登場します。ただし、Stripe単体が「ウォレット管理基盤」として出てくるというより、AWS AgentCore Paymentsでは `StripePrivy` というconnector typeとして、Privyのウォレット基盤と組み合わされて登場します。

### 14.5 Privy

Privyは、アプリにウォレット機能を組み込むためのウォレット/認証インフラです。

Privyを使うと、ユーザーがブラウザ拡張ウォレットを自分で用意しなくても、アプリ内でウォレットを作成したり、既存ウォレットを接続したりできます。

```text
Privyが主に提供するもの:
  ユーザー認証
  埋め込みウォレット
  サーバー管理ウォレット
  ウォレット署名
  権限やポリシー管理
```

### 14.6 Stripe + Privy

Stripe + Privyは、「Stripeの決済/金融インフラ」と「Privyのウォレット基盤」を組み合わせる文脈で登場します。

AWS AgentCore Paymentsでは、`StripePrivy` というconnector typeとして扱われます。

```text
Stripe:
  決済/金融インフラ側

Privy:
  ウォレット作成・署名・認証側

StripePrivy:
  AgentCore Paymentsが接続する外部ウォレット/決済プロバイダーの1つ
```

## 15. 全体の関係

ここまでを並べると、次の関係になります。

```text
Programmatic payments:
  ソフトウェアが支払いを扱う広い概念

マイクロペイメント:
  小さな金額の支払い

ステーブルコイン:
  プログラムから扱いやすいデジタルな支払い残高

Web3:
  ウォレット、ブロックチェーン、オンチェーン決済の技術領域

x402:
  HTTPで支払い要求と支払い証明をやり取りするプロトコル

Coinbase:
  x402の開発元

Coinbase CDP:
  ウォレットや暗号資産決済の開発者基盤

Stripe:
  決済/金融インフラ

Privy:
  ウォレット/認証インフラ

Stripe + Privy:
  決済インフラとウォレット基盤の組み合わせ
```

## 16. AI抜きの利用例

AIエージェントがなくても、こうした仕組みは使えます。

- 有料APIを1回だけ呼び出す
- 有料記事を1本だけ読む
- 有料データセットから1件だけ取得する
- ブラウザ拡張が必要な情報だけ購入する
- IoTデバイスが必要なときだけ外部データを買う
- 開発ツールが解析サービスを必要な回数だけ使う

つまり、中心にあるのはAIではなく「プログラムが有料リソースを必要な分だけ買う」という考え方です。

## 17. 注意点

実運用では、技術だけでなく以下の検討が必要です。

- 誰が支払いを許可したのか
- 1回あたり、1日あたり、1ユーザーあたりの上限はいくらか
- 誤作動や悪用で支出が膨らまないか
- 支払いログをどう監査するか
- 返金や失敗時の扱いをどうするか
- 会計、税務、法務、暗号資産ポリシーに合うか

## 18. まとめ

AWSやAIエージェント抜きで考えると、これは「HTTP/APIに少額決済を組み込む話」です。

その背景には、購入対象がデジタル化し、ソフトウェア同士がAPIやデータを細かく使うようになったことがあります。

ステーブルコインは支払い手段の一つで、x402は支払い要求と支払い証明をHTTP上でやり取りするための仕組みです。Coinbase CDPやStripe + Privyは、その支払いに必要なウォレットや署名を扱う外部基盤です。

## 19. 次の話へのつながり

ここまでの話で見えてくる課題は、次のようなものです。

```text
人間向けの決済:
  店舗、EC、サブスク、請求書には向いている

ソフトウェア同士の細かい購入:
  1回だけ、少額だけ、今すぐ使いたい、という場面では扱いづらい
```

AIエージェントは、この課題を解決する魔法の手段ではありません。むしろ、AIエージェントが普及すると、この課題がより目立つようになります。

理由は、AIエージェントがユーザーの依頼に応じて、外部のAPI、データ、ツール、コンテンツをその場で選んで使うからです。

```text
01:
  ソフトウェア同士の少額決済が必要になってきた

02:
  AIエージェントが出ると、その必要性がさらに強くなる

03:
  AWS AgentCore Paymentsは、その決済部分をAWSサービスとして扱うもの
```
